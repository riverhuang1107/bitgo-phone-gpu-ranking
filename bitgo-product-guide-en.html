bitgo

bitgo provides wallet-based billing, signed authorization, and reasoning-token consumption for agents while they build software, and for the software those agents create when it needs embedded model inference. It is not a single project-filtering tool; it is the payment and authentication layer that lets agent workflows and application runtime reasoning connect to model services reliably.

Designed for agents and embedded reasoning

Humans confirm the chain, address, amount, and authorization data; agents create wallet-aware signed requests and connect reasoning capability into development or product runtime flows.

1. Product Highlights

Payment Tool for the Agent EraBitGo provides decentralized virtual-asset payment capabilities for the Agent era, enabling AI agents, automation programs, and developer applications to consume resources safely and verifiably through cryptocurrency wallets. Instead of relying on a traditional account-balance system, it centers payment on wallet addresses, on-chain identity, and signed authorization, returning payment control to users and programs themselves.
Wallet as Payment IdentityIn BitGo, ETH, BTC, LTC, and other cryptocurrency wallets are not only asset holders. They also act as authorization credentials for reasoning APIs, model services, data services, and other digital resources. Each request uses a private-key signature to confirm identity and authorize spending, so the service does not need to custody user private keys or depend on centralized usernames and passwords.
Secure and Transparent Virtual-asset SpendingBitGo gives every resource-consumption event a clear source, quota, and record. Users can see which wallet initiated the request, how much balance was consumed, which fund ID it maps to, and complete usage details for token counts, unit prices, consumed amount, and remaining balance. This keeps automated agent payments auditable and verifiable.
From Token Spend to the Agent EconomyToken consumption is only the starting point for BitGo. The same wallet-signature and payment mechanism can expand to model calls, tool calls, data purchases, API subscriptions, compute rental, automated trading, task bounties, and cross-platform digital-service settlement, becoming a payment layer that connects wallets, assets, services, and agent-driven consumption.

Deposits are initiated by a human from an exchange or centralized wallet. The agent can generate instructions and verify parameters, but it must not guess the chain, amount, or receiving address.

  1. Choose the supported chain, such as BTC, LTC, or ETH, according to the service registration.
  2. Confirm the official receiving address and network with a human operator.
  3. Transfer the required amount from the exchange or centralized wallet.
  4. Wait for on-chain confirmations and record the deposit amount and ID.
  5. Use the registered wallet address, amount, and amount ID to sign API calls.

If the wrong chain or address is used, the agent cannot automatically recover the funds. Human confirmation remains mandatory for deposit-critical fields.

3. On-chain Address and Wallet Creation

bitgo identifies users by their on-chain address. One address can correspond to multiple wallets so different agents, software projects, environments, or business modules can isolate reasoning-token consumption.

Human-provided dataChain type such as ltc, btc, or eth, wallet address, deposit amount or denomination, money_id, and a private key injected through a secure channel.
Agent actionsCreate or configure wallet records, generate signed requests, test reasoning API connectivity, and connect model calls into development workflows or application runtime logic.

The chain type, amount, and authorization data must be explicitly provided by a human. Agents should not infer business amounts from address appearance, and real private keys must never be written into repositories, documentation, logs, or sample code.

Usage Fields

After each external reasoning API request completes, the server returns the complete usage JSON inside the response. The complete response example above already includes usage, so this section does not repeat a standalone usage JSON block.

The monetary fields in usage are integer-scaled values returned by the server. Convert them to actual amounts as follows:

  • The actual input_token_unit_price and output_token_unit_price values are the field value multiplied by 10^-5, and the unit price is the price per 1000 tokens.
  • The actual consume_amount value is the field value multiplied by 10^-8.
  • The actual balance value is the field value multiplied by 10^-8.

If the server response has no usage field, the CLI prints "usage": null; it does not estimate or fabricate usage.

This example comes from a real BTC Taproot wallet-signed request. X-Nonce, X-Signature, the wallet signature inside X-Params, balance, hash, and token counts change on each call. Private keys must not be written into docs, repositories, or logs.

6. Multi-chain Wallet Generation and Connectivity Tests

Cryptocurrency signing is handled by the Go signer. Agents or applications call the signer to assemble X-Params, then use a local ECDSA P-256 private key to sign X-Params + X-Nonce, and finally send the complete reasoning API request. Wallet generation and signing behavior differ by chain:

LTC
Generate a Litecoin mainnet wallet, export the address and WIF private key, and configure wallet_chain = "ltc". Normal addresses use compact ECDSA; Taproot addresses such as ltc1p... use Schnorr signing. LTC has been verified with real requests, but the chain type must still be explicitly provided by a human.
BTC
Generate a Bitcoin mainnet wallet, export the address and WIF private key, and configure wallet_chain = "btc". Normal addresses such as 1..., 3..., and bc1q... use compact ECDSA; Taproot addresses such as bc1p... use Schnorr signing with the tweaked private key.
ETH
Generate an Ethereum wallet, export the 0x... address and hex private key, configure wallet_chain = "eth", and let the signer generate authorization parameters using the Ethereum signing method.

7. Wallet-signed Reasoning API Calls

The reasoning API uses the wallet business signature in X-Params, plus X-Nonce, X-Signature, and X-Public-Key for interface-level ECDSA signing. The wallet-signature message is the direct concatenation of wallet address, amount, and amount ID:

${wallet_address}${money}${money_id}

X-Params Signing Process

  1. A human provides wallet_chain, wallet_address, money, money_id, and a securely injected private key.
  2. The agent concatenates wallet_address + money + money_id as the wallet-signature message.
  3. The signer hashes the message with sha256.Sum256 to get a 32-byte digest.
  4. ltc decodes the Litecoin mainnet WIF private key and chooses the signing method by address type: normal addresses use btcec/v2/ecdsa.SignCompact, while Taproot addresses such as ltc1p... use btcec/v2/schnorr.Sign.
  5. btc decodes the Bitcoin mainnet WIF private key. Normal addresses such as 1..., 3..., and bc1q... use btcec/v2/ecdsa.SignCompact. Taproot addresses such as bc1p... first call txscript.TweakTaprootPrivKey(*privateKey, []byte{}) to tweak the internal private key, then use btcec/v2/schnorr.Sign.
  6. eth uses an Ethereum hex private key, with or without the 0x prefix, and calls github.com/ethereum/go-ethereum/crypto.Sign.
  7. Base64-encode the wallet signature and assemble a JSON object with wallet_address, money, money_id, and signature.
  8. Base64-encode that JSON object and place it in the HTTP X-Params header.
  9. Generate a random X-Nonce, concatenate X-Params + X-Nonce, and hash it with SHA-256.
  10. Use the local ECDSA P-256 private key to generate an ASN.1/DER signature hex, write it to X-Signature, and write the local ECDSA public key DER hex to X-Public-Key.
FieldDescription
wallet_addressOn-chain address, also the key identity used by bitgo
moneyHuman-provided amount or denomination; agents should not guess it
money_idAmount record or wallet record ID used by the server
signatureBase64 string generated by signing wallet_address + money + money_id

After the signature fields are assembled as JSON, the JSON string is base64-encoded into the request header, and the interface-level request signature is generated:

{
  "wallet_address": "YOUR_WALLET_ADDRESS",
  "money": "YOUR_WALLET_MONEY",
  "money_id": "YOUR_MONEY_ID",
  "signature": "BASE64_SIGNATURE"
}

9. Security Notes

  • Do not commit real private keys, complete X-Params, complete X-Signature, X-Public-Key, or management tokens to the repository.
  • Chain type, deposit network, amount, and money_id must be confirmed by a human before an agent uses them.
  • Private keys should be injected through environment variables, secret managers, or deployment platforms, not stored in templates.
  • Always verify the official receiving address and mainnet before depositing. Transfers to the wrong address or network cannot be recovered by agents or software.
  • Applications should record usage for cost tracking, but should not log sensitive headers or private-key material.
